## 2. Stage commits, logically grouped

**Commit A — audio survival under route change**
```bash
git add Daisy/AudioRecorder.swift Daisy/AudioInputDevices.swift
git commit -m "Audio: survive wired headphone plug-in mid-recording

applyPreferredInputDevice no longer short-circuits when the user has
'System default' picked — always resolves to a concrete AudioDeviceID
and binds AUHAL explicitly. Stops the silent-recording-after-route-change
failure where the unit stayed bound to a device macOS had invalidated.

Adds a CoreAudio listener on kAudioHardwarePropertyDefaultInputDevice
to catch route changes AVAudioEngineConfigurationChange misses (output-
only headphones on macOS 14+). Single debounce point inside
handleConfigurationChange so NC + CoreAudio firing for the same event
don't double-restart.

Adds a 5-second recovery watchdog that falls the session to .paused
with a Resume toast if no audio buffer arrives post-restart — covers
the 'engine reports started but tap is dead' case the previous code
never noticed."
```

**Commit B — calendar auto-stop with manual hotkey start**
```bash
git add Daisy/RecordingSession.swift
git commit -m "Calendar: auto-stop fires for manually-started sessions

scheduleAutoStopIfNeeded() silently returned when the user hit the
record hotkey before CalendarService.tick() got to auto-start — no
boundMeeting, no auto-stop, four distinct failure modes that all
looked identical from Console.

New bindCurrentMeetingIfPossible() attaches the session to a
calendar event currently in its -120s…+30s start window. Mirrors
the CalendarService.tick() fire window so any meeting we'd have
auto-started is also a meeting we'll auto-bind to. Title overwrite
guarded by an exact regex match on the autogenerated
'Meeting YYYY-MM-DD HH:MM' placeholder so user-typed titles
(including 'Meeting with Anna prep') survive.

Adds dedicated Logger(category: 'AutoStop') and logs every early-
return in scheduleAutoStopIfNeeded() so the next instance of this
class of bug is 30 seconds of \`log stream\` away, not invisible.
Re-arms on resume() as a late-binding safety net."
```

**Commit C — Calendar settings UI move + permission sync**
```bash
git add Daisy/SettingsView.swift Daisy/ConnectionsView.swift Daisy/MainView.swift Daisy/HomeView.swift
git commit -m "Settings: move Calendar behaviour toggles from Connections to General

Calendar permission status (Granted/Denied/etc.) already lives in
Settings → Permissions; the behaviour toggles (auto-start, auto-stop,
menu-bar next-meeting, grace period) belong next to the other
recording prefs in Settings → General, not under Connections which
is now reserved for external integrations only.

hasAnyCalendarSource in Settings, HomeView, and the gating across
the three surfaces now reads SystemPermissions.calendar (the same
observable PermissionsView watches) instead of the
settings.calendarAccessGranted UserDefaults cache. Pre-1.0.4 the
cache could lag behind the live state, which is what a tester hit:
Permissions tab said Granted, General tab showed disabled toggles.

ConnectionSection enum loses .calendar (no deep-link callers). The
Google OAuth backend stays alive but its UI is dormant pre-
verification — when Google approves the verification questionnaire
the OAuth row comes back as its own tab next to Notion/MCP server."
```

**Commit D — release.sh build sanity check**
```bash
git add scripts/release.sh
git commit -m "release.sh: fail fast when build <= last published

New [0/6] step parses every <sparkle:version> in appcast.xml, takes
the max, and refuses if the supplied build is <= that value. Catches
the class of bug that shipped 1.0.3 build 6 alongside the already-
published 1.0.2 build 7, which Sparkle correctly served as a
downgrade to the tester. Failing here costs < 1 second; failing in
step [6/6] after archive + notarize costs 15 minutes.

pbxproj's CURRENT_PROJECT_VERSION is stale (release.sh doesn't bump
it), so the authoritative source for 'next free build' is appcast.xml
itself."
```

produce loud diagnostic output. Silent failure for the same root
