# Daisy 1.0.5.3 — MCP server reliability

## Fixed

- **MCP connection no longer hangs after the first AI request.** The Daisy MCP server's SSE stream had no keepalive — once idle for ~30 seconds (you finishing one Claude prompt and starting another), the loopback socket would silently go half-open. The next prompt's request would land, Daisy would write the response into a TCP send buffer that would never drain, and the AI client (Claude Desktop via the mcp-remote bridge) would hang for 3+ minutes. Now the server emits an SSE comment-frame ping every 15 seconds for the lifetime of the connection, well under the staleness window MCP clients use.
- **Claude Desktop reload after a hung session works correctly.** Previously, a hang would leave the bridge in a bad state; on next Claude launch it'd refuse to load Daisy with "not valid MCP server configurations: daisy". The session-id correlation + clean SSE teardown rolls the stale state forward on reconnect.
- **"Add to Claude Desktop" button now writes a configuration that works on the first try.** The shipped snippet (and the in-app installer) now passes `--transport sse-only --allow-http` to `mcp-remote`. Without them, the bridge tried both Streamable HTTP and SSE in sequence and refused plain HTTP on the 127.0.0.1 loopback. Existing manual configs need the same two flags added — see the updated [docs/mcp page](https://mydaisy.io/docs/mcp).

## Internals

- New `sseKeepaliveTimer` dispatch source on `MCPServer`, scoped to the lifetime of the active SSE connection. Re-armed on every fresh stream, cancelled on stop / teardown / replacement.
- `Mcp-Session-Id` header emitted on the SSE `200 OK` response. Per-connection UUID; logged so a hung POST can be correlated with the stream that should have answered it.
- Race fix in `handlePostedMessage`: the SSE reference is snapshotted before the async tool dispatch and re-validated (`sseConnection === sseAtEntry`) before writing the response. A fresh client reconnect rolling the stream over mid-request now drops the stale response with a log warning instead of writing to a cancelled connection.
- Single `tearDownSSE()` chokepoint replaces three scattered cancel+nil sites — keepalive timer and session id are torn down atomically with the connection so a fire-after-cancel race can't leak.
