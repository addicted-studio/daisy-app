# Daisy 1.0.7.1 — Silent audio-write death fix + post-stop telemetry

## Fixed

- **Persistent "Markdown copied to clipboard" banner across session switches.** Pre-fix the in-detail status banner stored its state in a SwiftUI `@State` belonging to `SessionDetailView`. NavigationSplitView reuses the detail view across selection changes, so a "Markdown copied" success message you triggered on session A still showed when you clicked into session B (and session C, and so on) until you manually dismissed it with the ⨯. Moved every action-feedback callsite — move-to-folder, re-summarize, copy markdown — onto the existing `ToastCenter`, which renders globally above all windows and self-dismisses after a few seconds. Same channel the new per-block copy buttons (above) already used, so action feedback now lives in one place with one lifecycle.

- **Silent AVAudioFile write-death surfacing.** Pre-fix failure mode caught during a 2026-05-25 internal test: SCStream/AVAudioEngine kept delivering buffers to the live transcribe pipeline (Whisper saw the full session and produced a complete transcript), but the on-disk `.caf` archives stopped growing after some mid-session disruption. The user's `system_audio.caf` landed as 0 bytes while the transcript covered 44 minutes; `microphone.caf` was truncated to 5% of session length. There was zero indication anything had failed — the frontmatter even stamped `daisy_system_audio_status: captured` because the in-memory `hasReceivedAudio` flag was still true (it flips on the first SCK buffer regardless of disk-write outcome). 1.0.7.1 adds a render-thread frame-write counter in both `AudioRecorder` and `SystemAudioCapture`, cross-checks it against the on-disk file size and the write-error counter at stop time, and surfaces a loud toast plus a new `truncated` frontmatter state when the disk write didn't actually persist the audio. Users now know their session is partial before they let delete-after-transcription nuke the source files.

## Added

- **Two-block collapsible layout in Session detail.** The session view now reads as two collapsible cards instead of a flat scroll of mdSection cards: outer **Summary** block (Meeting lede + topical sections + Next actions + Follow-up + screenshots) and outer **Transcript** block. Each block has its own header chevron and its own copy-to-clipboard button on the trailing edge — copy emits plain text (no markdown noise) ready to paste into Slack, email, or a chat agent. Expanded/collapsed state for each block persists per-user via UserDefaults — users who live in transcripts and rarely scroll the summary (or vice versa) don't have to re-collapse every time they open a session.

- **Native full-content selection on the transcript body.** Pre-fix on 1.0.7 a drag-select inside the transcript only captured the visible viewport — anything that had scrolled out of view couldn't be reached, and ⌘A was similarly clipped. New `SelectableTextView` wraps a read-only `NSTextView` for the transcript body so the full text lives in the AppKit text system, which has had correct full-content selection, ⌘A, ⌘C, and ⌘F (Find Bar) for decades. Drag-select now extends through the whole transcript regardless of scroll position; ⌘F opens a Find Bar over the text. Summary block keeps the existing SwiftUI `Text(...).textSelection(.enabled)` rendering since each inner card is short enough that the SwiftUI selection model handles it correctly.

- **Per-stage timing logs across the post-stop pipeline.** Every stage between "user hit Stop" and the summary task spawning now emits a `post-stop <stage>: Xms` line plus a matching `OSSignpost` interval under category `PostStop` (subsystem `app.essazanov.Daisy`). Stages: `final_transcribe_mic`, `final_transcribe_system`, `speaker_profile_match`, `render_markdown`, `write_transcript_md`, `session_store_refresh`, then the existing `summarize` / `write_summary` / `auto_send` in the detached finalize task. Closes the silent-stages gap that made earlier "post-processing is slow" reports impossible to triage: any user can now run `log show --predicate 'subsystem == "app.essazanov.Daisy"' --last 10m --info --debug` (or System Trace → Signposts in Instruments) and see exactly which stage is eating time.

- **Smarter Name-the-speakers UX.** Rows in the speaker-mapping card are now sorted by talk-time (most segments first), so the dominant voice in the meeting is the first row the user sees — pre-fix the ordering was alphabetical, which put a 17-segment "Remote A" above a 380-segment "Remote B". Each row also shows its segment count as a subtle caption. When a transcript cluster has no centroid in `speakers.json` (FluidAudio assigned the label but didn't form a clean voice fingerprint — surfaced in the 2026-05-25 tester Sync session where Remote D had 17 segments but no centroid), the row gets a small "session only" pill explaining the rename applies to this session's display only and won't auto-match future recordings. Doesn't disable the input — naming a session-only fragment is still useful for transcript readability.

- **`daisy_mic_audio_status` frontmatter line.** Symmetric to the existing `daisy_system_audio_status`. Same four states (`off` / `captured (bytes B)` / `empty` / `truncated (...)`). Lets a folder of sessions be grepped for partial-mic incidents the same way system-audio incidents already could be.

## Changed

- **`daisy_system_audio_status` frontmatter format.** Now includes byte count for `captured`/`truncated` states, e.g. `captured (4823041 B)` or `truncated (0 B on disk, 0 frames written, 3 write errors)`. Previously bare `captured` / `empty`. The four-state taxonomy (`off` / `captured` / `empty` / `truncated`) replaces the prior three-state one — `truncated` is new and names the failure mode 1.0.7 shipped without a name for.

## Why now

PH launch moved from 28 May → 2 June 2026 specifically to ship this fix end-to-end instead of the cosmetic-only frontmatter cross-check that would have been the time-boxed alternative. Silent audio loss is the worst kind of bug for a meeting-recorder app — the user trusts that whatever they hit Record on will be there afterwards. Even if the underlying CoreAudio/SCKit failure modes that triggered the truncation are out of scope to fix in 1.0.7.1 (deferred to 1.0.8 investigation), the user now at least KNOWS when their audio is partial, and can keep the raw `.caf` for forensic repair instead of trusting a lying frontmatter.
